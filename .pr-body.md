## Bug

When a player disconnected mid-question and reconnected, `computeResumedPossessionTiming` overwrote `shownAt` with `resumedAt` whenever the question was already revealed (`revealRemainingMs === 0`). This reset the scoring clock (`nowMs - shownAtMs`) to ~0, handing the reconnecting player a **fresh full-duration window** and an **inflated clue index / points payout**.

## Repro (from a real bug report — match `01e47688`)

| | Buggy (current) | Fixed |
|---|---|---|
| q11 shown_at | overwritten to reconnect time | preserved at original |
| Player reconnects | 36s into a 50s clue_chain | same |
| Answers 2.2s later | timeMs ~ 2.2s -> **100 pts** (clueIndex 0) | timeMs ~ 38.4s -> **40 pts** (clueIndex 3) |

The player who was present the whole time and answered legitimately in 4.7s (100 pts) was cheated out of a fair race — the reconnecting opponent got the same 100 pts despite being absent for 36s.

## Fix

In `computeResumedPossessionTiming` (`possession-timing.ts`): when the reveal already completed (`revealRemainingMs === 0`, i.e. `shownAt <= pauseStart`), preserve the **original** `shownAt` for scoring. Only the `deadlineAt` shifts to preserve the remaining answer time. The "not yet revealed" path (`shownAt` in the future) is unchanged.

```diff
-  return {
-    playableAt: new Date(params.resumedAtMs + revealRemainingMs),
+  const playableAtMs = revealRemainingMs > 0
+    ? params.resumedAtMs + revealRemainingMs
+    : shownAtMs;
+  return {
+    playableAt: new Date(playableAtMs),
     deadlineAt: new Date(params.resumedAtMs + answerRemainingMs),
   };
```

## Tests

- Updated two existing tests that pinned the buggy `playableAt = resumedAt` behavior to expect the preserved original `shownAt`.
- Added a regression test reproducing the bug-report scenario (5-clue question, 36s disconnect) — verifies `timeMs ~ 38.4s` -> `clueIndex 3` -> 40 pts (NOT 100).
- Full possession suite: **77/77 pass**, no regressions.
- `npm run lint` (tsc --noEmit): clean.

## Impact on the reported match

In match `01e47688`, the bug gave Luka_Ivan 100 pts instead of 40 on the last question. However, even with the corrected 40 pts the possession bar would have ended at -90 (kakhaL needs <= -100 for a goal), so the match outcome (1-0 Luka win, +50/-25 RP) is **unchanged**. No RP correction is owed for this match, but the bug could flip results in other scenarios and must be fixed.
