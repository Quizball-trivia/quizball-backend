# Streamer-night database incident: findings and capacity proof

Updated: 2026-07-15. Scope: local and staging only. Production was not modified
or load tested.

## Bottom line

`EDBHANDLEREXITED: connection to database closed` is a symptom: a Postgres
connection already held by the Node process disappeared because the pooler,
database, or network closed it. The retained logs do not prove which upstream
event initiated the original production burst. A Supabase maintenance/network
event remains possible, and the previous 30-connections-per-replica budget made
pool exhaustion plausible, but neither claim is conclusive without the expired
Supavisor event logs.

What is proven is why the application turned a connection-closure burst into a
long outage: two replicas could over-claim the shared DB budget, requests could
wait behind unhealthy work, health checks did not prove DB readiness, and a
sick process was allowed to stay alive. Those failure amplifiers are fixed.

The investigation also found two independent staging/load-test problems that
would have produced misleading conclusions:

1. Railway staging ran in Virginia while Supabase ran in Frankfurt. A trivial
   DB readiness query took roughly 359–375 ms even though Postgres execution
   times were mostly below 8 ms.
2. A single load generator is one source IP. Supabase Auth correctly allows a
   `/token` burst of 30 per IP, so 100–5,000 login tests from one Mac measure the
   Auth abuse guard, not real-user capacity.

Staging backend and Redis now run in Railway EU West (Amsterdam), near the
Frankfurt database. DB readiness dropped to 33–41 ms without increasing the
12-connection-per-replica budget.

## Fixes now on staging

- `DB_POOL_MAX=12` and `DB_INFLIGHT_LIMIT=12` per replica; two replicas use at
  most 24 app connections.
- `DB_QUEUE_LIMIT=48` and 1.5 s acquisition deadline; overload sheds with a
  retryable 503 instead of hanging or returning a raw driver 500.
- Active DB watchdog and real `/health/db`; repeated probe failure exits so
  Railway replaces a poisoned replica.
- Process/socket rejection boundaries so detached async failures cannot leave a
  half-alive process.
- Stale matchmaking entries are removed and skipped instead of blocking the
  queue.
- Simultaneous gameplay answers wait briefly for the per-match lock instead of
  dropping an answer as `MATCH_BUSY`.
- Supabase Auth IP forwarding is enabled with a modern staging-only secret key;
  real clients no longer collapse onto the two Railway replica IPs.
- Geo fallback lookups are coalesced, positively and negatively cached, and
  bounded to 500 ms. An unavailable geo provider can no longer pin every
  authenticated endpoint at a 3 s timeout.
- The gameplay harness reuses valid tokens between matches/reconnects and paces
  initial single-host session creation. Login/refresh load remains a separate
  distributed k6 test.

## Red -> green evidence

### Local incident reproduction

- Old behavior under a DB blackhole/reset: p95 about 6.7–6.9 s, p99 about
  7.6–7.8 s, with raw connection-closed 500s.
- New behavior under the same fault: only bounded retryable 503s at about the
  1.5 s admission deadline, followed by recovery.
- Direct `pg_terminate_backend` at 120 RPS: 3,600/3,600 responses completed with
  no server errors.
- Three failed watchdog probes caused exit 1; the supervisor restarted the
  process and DB health returned 200.

### Staging 100-player red run (Virginia -> Frankfurt)

Report: `reports/staging-capacity-100-clean.json`.

- App pools reached active 12 + queued 48 on both replicas.
- 648 DB admission sheds and four DB readiness failures.
- Only 55/59 eligible matches completed before the deadline.
- Peak Postgres connections were only 37/60 and hot queries were fast. The first
  wall was network round-trip amplification and the app admission queues, not
  Postgres execution or CPU.
- This run also exposed invalid one-IP 429s because the local harness did not
  have staging's limiter bypass; those 429s are not counted as capacity proof.

### Staging 100-player gameplay/database green run (Amsterdam -> Frankfurt)

Report: `reports/staging-capacity-100-eu-green.json`.

- 13,500/13,500 HTTP requests completed with status 200; zero server errors.
- 164/164 eligible matches completed.
- Zero forfeits, abandons, deadline cutoffs, boot-stage violations, or socket
  errors.
- Zero DB admission rejections/timeouts and zero readiness failures.
- Worst observed app admission wait: 63.2 ms.
- Peak Postgres connections: 34/60 (56.7%); peak active: 10; no lock waits;
  longest active query: 0.1 s.
- CPU peak by replica: 41.1%; event-loop p99: 26.6 ms.
- HTTP latency still failed because missing-country users repeatedly waited on
  the external geo service. This led directly to the geo-cache fix above.

### Staging 100-user HTTP geo red -> green gate

Report: `reports/staging-http-100-geo-green.json`.

- 5,750/5,750 requests completed, all status 200.
- Zero unexpected 4xx, 5xx, DB sheds, or DB timeouts.
- Slowest route p95: 835 ms (threshold 1.5 s).
- Slowest route p99: 1.725 s (threshold 3 s).
- Peak Postgres connections: 31/60 (51.7%); no lock waits.
- Formal SLO verdict: **PASS**.

k6 smoke also passed the real login -> `/users/me` -> wallet -> refresh ->
`/users/me` journey with 100% checks, login p95 332 ms, refresh p95 154 ms,
wallet p95 285 ms, and no server errors.

## Capacity statement (do not overclaim)

The current non-production evidence supports this statement:

> Green at 100 concurrent gameplay clients plus a 50 RPS weighted 33-route API
> mix on two Railway replicas, with a 12-connection DB budget per replica.

It does **not** yet prove 500, 1,000, 2,000, or 5,000 real users on staging.
One Mac can drive 500 local gameplay clients, but real Auth login/refresh tests
at those levels require multiple source IPs. A trustworthy 5,000-user number
also requires staging Supabase/Railway compute to match the intended production
topology; otherwise the result measures the smaller staging tier.

## Required next campaign for 500 -> 5,000

1. Run free k6 plus the Socket.IO fleet from 2–5 short-lived EU VPS workers,
   each with a disjoint user shard and source IP.
2. Ladder: 100, 500, 1,000, 2,000, 5,000. Hold each rung for five minutes and
   stop at the first SLO failure.
3. At the highest passing rung, run a 60-minute soak and a separate reconnect
   chaos run across search, draft, gate, and active-match stages.
4. During a passing rung, restart one staging replica and terminate staging DB
   pool connections once. Require bounded 503s, healthy-replica continuity, and
   recovery in under 60 seconds.
5. Operate at no more than 50–60% of the twice-confirmed ceiling.

Paid Grafana Cloud k6 is optional. Free k6 on short-lived VPS workers is enough;
the remaining requirement is approval for the small VPS/tier cost, not another
application-code change.
